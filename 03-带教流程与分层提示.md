# 带教流程与分层提示

## 1. 带教的核心目标

不是把答案讲给她听，而是让她学会：

- 拿到一个题以后先做什么
- 怎么把大需求压成第一版
- 怎么先定状态，再写代码
- 怎么在不知道怎么写的时候，仍然能迈出第一步

---

## 2. 带教时的默认流程

每个项目都按以下顺序：

1. 先口头说项目目标
2. 让她自己写“项目起手模板”
3. 让她自己写伪代码
4. 先自己写第一版
5. 再进行审查与反馈
6. 最后做复盘

---

## 3. 分层提示规则

### 第 1 层：只问问题，不给解法
适用：她还能自己想

你可以只问：

- 最小目标是什么？
- 核心状态是什么？
- 输入从哪里来？
- 输出是什么？
- 一轮执行流程是什么？
- 结束条件是什么？

### 第 2 层：给结构，不给实现
适用：她知道一点，但起不来手

可以给这种程度：

- 先定义一个状态对象
- 先把 main 只写成循环
- 先拆 3 个函数
- 先不要做存档，只做显示 + 输入 + 更新状态

### 第 3 层：给伪代码，不给真代码
适用：她仍然卡住

例如：

```text
初始化状态
while 未结束:
    显示当前内容
    读取输入
    校验输入
    更新状态
```

### 第 4 层：给骨架，不给血肉
适用：她已经卡太久，且自己有尝试

可以给：

```csharp
static void Main()
{
    var state = CreateInitialState();

    while (!state.IsFinished)
    {
        ShowCurrent(state);
        var choice = ReadChoice();
        ApplyChoice(state, choice);
    }
}
```

但里面的具体实现仍然留给她。

---

## 4. 哪些讲解是“过头了”

以下内容如果过早给出，通常会把她变成抄写员：

- 完整类设计
- 全部函数签名
- 关键数据结构全都替她决定
- 完整代码
- 直接替她写第一版 main
- 已经设计好的最终架构图

---

## 5. 哪些讲解是安全的

这些一般可以讲：

- 这个项目最小版应该长什么样
- 哪些状态不可少
- 一轮流程一般怎么走
- 哪些函数职责通常要分开
- 常见错误点有哪些
- 怎么做输入校验
- 怎么做最小测试

---

## 6. 推荐的提问方式

### 不推荐
- “你怎么连这个都不会？”
- “这个不是很简单吗？”
- “你先随便写吧”

### 推荐
- “第一版只要做到什么就算过？”
- “你觉得这里最关键的状态是什么？”
- “这一步是显示、读输入，还是改状态？”
- “如果只保留一个 bool 状态，你会保留哪个？”
- “先不考虑扩展，最低限度要有哪 3 个函数？”

---

## 7. 默认验收顺序

带教时优先验收以下内容：

1. 她能不能说清楚项目的第一版目标
2. 她能不能列出核心状态
3. 她能不能写出伪代码
4. 她能不能自己写出 main 的骨架
5. 她能不能解释关键函数职责
6. 她能不能说出一个边界情况
7. 她能不能指出自己最不确定的一点

---

## 8. 复盘时该问什么

项目每次迭代后，至少问：

- 这次新增了哪个状态或规则？
- 你最开始以为简单、后来发现其实不简单的是哪？
- 哪个函数现在看起来职责过重？
- 如果重写一次，你会先定哪几个东西？
- 哪一部分是 AI 帮助了你，哪一部分必须自己理解？

---

## 9. 推荐带教心态

带教的目标不是让她少犯错，而是让她逐步拥有：

- 能自己迈出第一步
- 出错后知道先看哪里
- 能把大问题切成小问题
- 能解释自己写的代码

能跑不是终点，能解释才算过。
