# 编程练手项目文档包

这是一套给“**会照着现有模块改，但一离开完整壳子就不知道怎么下手**”的学习者准备的练手文档。

目标不是做出炫酷项目，而是训练下面这些能力：

- 把一个“大东西”压成“第一版最小可运行”
- 先定义状态、输入、输出，再开始写代码
- 理解一轮执行里到底发生了什么
- 区分“逻辑骨架”和“表面文本/素材”
- 在 AI 参与时，仍然保留自己的建模和实现主导权
- 形成提交、复盘、验收、代码评审的基本习惯

## 建议使用顺序

1. 先读 `01-通用训练规则.md`
2. 再读 `02-AI使用协议.md`
3. 再读 `03-带教流程与分层提示.md`
4. 练手时配合：
   - `04-提交与验收标准.md`
   - `05-AI审查提示词模板.md`
   - `templates/项目起手模板.md`
   - `templates/提交模板.md`
5. 然后按顺序做 `projects/` 里的项目

## 推荐顺序

1. 文字版任务 / 对话状态机
2. 背包 + 商店 + 物品效果模拟器
3. 回合制战斗模拟器
4. 存档系统 + 版本迁移工具
5. 小型规则引擎 / 条件判定器
6. 简单布局求解器 / 安全区模拟器
7. C++ 资源包装器 / 小日志系统

- `01-通用训练规则.md`：练手目的、约束、默认工作方式

- `03-带教流程与分层提示.md`：带教时的提示层级
- `04-提交与验收标准.md`：每次提交最少要带什么

## 关于 AI 生成文本素材

默认**允许** AI 生成以下内容：

- NPC 对话文本
- 道具描述
- 任务说明
- 世界观背景文字
- 菜单文案
- 测试数据中的文本部分
- 示例存档中的文本部分

但要遵守两个限制：

1. **文本素材不能替代逻辑设计。**
   AI 可以写“剑的描述”，但不能替她决定“剑的效果系统怎么建模”。

2. **文本素材要晚于结构。**
   先把状态、字段、规则、接口、函数职责定下来，再补文案。

## 一个简单判断标准

如果一个问题决定的是“程序怎么组织、状态怎么流、代码为什么这样写”，那不能直接让 AI 代做。
如果一个问题决定的是“这个 NPC 说什么更自然、这个物品描述更像游戏文案”，那可以交给 AI。
